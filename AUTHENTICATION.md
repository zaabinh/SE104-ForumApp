## Purpose

## Authentication Architecture

The system uses JWT authentication with:

- Access token for authenticated API access
- Refresh token for renewing access without logging in again
- Persistent refresh session storage in the database

## Token Model

### Access token

Generated by:

- `create_access_token(user_id, email, role)`

Payload fields:

- `sub`: user id
- `email`: user email
- `role`: user role
- `type`: `"access"`
- `exp`: expiration timestamp

Lifetime:

- Controlled by `ACCESS_TOKEN_EXPIRE_MINUTES`
- Default: `60` minutes

Usage:

- Sent in the HTTP header:

```text
Authorization: Bearer <access_token>
```

### Refresh token

Generated by:

- `create_refresh_token(user_id)`

Payload fields:

- `sub`: user id
- `type`: `"refresh"`
- `exp`: expiration timestamp

Lifetime:

- Controlled by `REFRESH_TOKEN_EXPIRE_DAYS`
- Default: `7` days

Persistence:

- Stored in table `auth_sessions`
- Linked to a specific user
- Stores:
  - refresh token value
  - IP address
  - user agent
  - expiration timestamp

## Password Handling

Password hashing is implemented in [hash.py](/d:/ZB/Code/UIT/NMCNPM/SE104.ForumApp/backend/utils/hash.py).

Current behavior:

- Passwords are hashed with Argon2 via `passlib`
- Password verification uses the stored Argon2 hash

Functions:

- `hash_password(password: str) -> str`
- `verify_password(password: str, hashed: str) -> bool`

This is the current expected dependency stack:

- `passlib[argon2]`

## Authentication Endpoints

Base prefix:

```text
/auth
```

### 1. Register

Endpoint:

```text
POST /auth/register
```

Request schema:

```json
{
  "username": "student123",
  "email": "student@example.com",
  "password": "secret123",
  "full_name": "Nguyen Van A"
}
```

Validation rules:

- `username`: 3 to 50 characters
- `email`: valid email address
- `password`: 6 to 128 characters
- `full_name`: 2 to 255 characters

Behavior:

- Normalizes email to lowercase
- Trims username
- Rejects duplicate email
- Rejects duplicate username
- Hashes password before storing
- Creates a new user with:
  - `role = "Student"`
  - `status = "Active"`
  - `provider = "local"`

Success response:

```json
{
  "message": "User registered successfully."
}
```

Common failure cases:

- `400`: email already exists
- `400`: username already exists
- `500`: database failure during registration

### 2. Login

Endpoint:

```text
POST /auth/login
```

Request schema:

```json
{
  "identifier": "student@example.com",
  "password": "secret123"
}
```

Behavior:

- Accepts either email or username in `identifier`
- Searches user by:
  - lowercase email
  - lowercase username
- Rejects if user not found
- Rejects if account status is not active
- Verifies password hash
- Generates:
  - access token
  - refresh token
- Stores refresh session in `auth_sessions`

Success response:

```json
{
  "access_token": "<jwt>",
  "refresh_token": "<jwt>",
  "token_type": "bearer",
  "expires_in": 3600
}
```

Common failure cases:

- `401`: user not found
- `401`: invalid password
- `403`: inactive account

### 3. Refresh Token

Endpoint:

```text
POST /auth/refresh
```

Request schema:

```json
{
  "refresh_token": "<jwt>"
}
```

Behavior:

- Looks up refresh token in `auth_sessions`
- Rejects if session does not exist
- Rejects and deletes session if expired
- Decodes JWT and verifies:
  - token is valid
  - token `type` is `"refresh"`
  - token subject matches session user
- Loads user
- Creates a new access token
- Returns the same refresh token if still valid

Success response:

```json
{
  "access_token": "<jwt>",
  "refresh_token": "<same-refresh-token>",
  "token_type": "bearer",
  "expires_in": 3600
}
```

Common failure cases:

- `401`: refresh token not found
- `401`: refresh token expired
- `401`: invalid token
- `401`: invalid token type
- `401`: user not found

### 4. Logout

Endpoint:

```text
POST /auth/logout
```

Request schema:

```json
{
  "refresh_token": "<jwt>"
}
```

Behavior:

- Deletes matching refresh session from `auth_sessions`
- Returns success even if the token was already missing

Success response:

```json
{
  "message": "Logged out successfully."
}
```

### 5. Current User

Endpoint:

```text
GET /auth/me
```

Authentication:

- Requires bearer access token

Behavior:

- Uses `get_current_user`
- Returns the authenticated user profile

Success response fields:

- `id`
- `username`
- `email`
- `full_name`
- `avatar_url`
- `bio`
- `role`
- `status`
- `provider`

### 6. Get User By ID

Endpoint:

```text
GET /auth/users/{user_id}
```

Authentication:

- Requires bearer access token

Behavior:

- Returns user profile by database id
- Uses authenticated user dependency for access control

## Dependency-Based Authentication

File:

- [backend/dependencies/auth.py](/d:/ZB/Code/UIT/NMCNPM/SE104.ForumApp/backend/dependencies/auth.py)

### `get_current_user`

Responsibilities:

- Reads `Authorization` bearer token
- Decodes JWT
- Validates token type is `access`
- Loads user from database
- Rejects inactive users
- Returns `User` model instance

Failure cases:

- Missing credentials
- Invalid or expired token
- Invalid token type
- Missing subject
- User not found
- Inactive account

### `require_role(*roles)`

Responsibilities:

- Builds a role-based dependency
- Ensures current user role matches one of the allowed roles
- Returns the authenticated user if authorized

Current state:

- Available for authorization logic
- Not heavily used yet in the current route set

## Frontend Authentication Flow

Frontend file:

- [frontend/lib/axios.ts](/d:/ZB/Code/UIT/NMCNPM/SE104.ForumApp/frontend/lib/axios.ts)

Stored local keys:

- `access_token`
- `refresh_token`
- `auth_user`

Main frontend functions:

### `saveAuthSession(tokens, user?)`

- Stores access token
- Stores refresh token
- Optionally stores user profile

### `fetchCurrentUser()`

- Calls `GET /auth/me`
- Stores the returned user object in local storage

### `refreshAccessToken()`

- Calls `POST /auth/refresh`
- Updates local tokens if refresh succeeds
- Clears auth state if refresh fails

### Axios request interceptor

- Automatically adds bearer access token to outgoing API requests

### Axios response interceptor

- Watches for `401`
- Attempts one refresh retry
- Replays the original request if refresh succeeds

### `logout()`

- Calls `POST /auth/logout`
- Clears local auth state

## Security Notes

Current strengths:

- Passwords are hashed with Argon2
- Access and refresh tokens are separated
- Refresh sessions are persisted server-side
- Inactive users are blocked

Current limitations:

- Refresh token rotation is not implemented yet
- Local storage is used instead of HttpOnly cookies
- Login rate limiting is not implemented yet
- Audit logging is minimal
- Default JWT secret fallback must not be used in production

## Recommended Next Improvements

1. Rotate refresh tokens on every refresh.
2. Move tokens to secure cookies if browser architecture allows it.
3. Add login throttling and brute-force protection.
4. Add email verification and password reset.
5. Add auth-specific tests for register, login, refresh, logout, and protected routes.
